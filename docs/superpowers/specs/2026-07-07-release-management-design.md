# Release management + README polish — design

**Goal:** introduce automated, semantic-commit-driven release management for
`claude-core` (GitHub Releases + CHANGELOG, no PyPI artifact) and rewrite
`README.md` to match the polish level shipped for `downbeat`.

**Context:** `claude-core` is not a packaged Python project — no
`pyproject.toml`/`setup.py` exists today. It's a bash-installed toolkit
(`install.sh`/`bootstrap.sh` symlinking skills into `~/.claude/skills`) plus a
native Claude Code plugin (`.claude-plugin/plugin.json`, name
`claude-core-hooks`, version hardcoded `"0.1.0"`). No git tags or GitHub
Releases exist yet. CI today is only `.github/workflows/portability.yml`
(pytest + ruff matrix, ubuntu/macos × python 3.11/3.13). Commit history is
partially conventional-commit-clean: the last 9 commits (since the Phase-2
plugin migration) use `feat:`/`fix:`/`docs:`/`ci:` properly; earlier history
uses ad-hoc `[R2]`/`[R3]` bracket-tag prefixes.

This design follows the OSS release-prep playbook
(`~/knowledge/wiki/insights/OSS release prep playbook.md`, derived from the
`downbeat` OSS-prep session) — this task covers only the release-management +
README slice of that playbook. Community-health files (LICENSE, CONTRIBUTING,
CODE_OF_CONDUCT, SECURITY) and making the repo public are explicitly deferred
to a future "go public" task — not part of this spec.

## 1. Version source of truth

- `python-semantic-release`, same tool as `downbeat`, but with **no PyPI
  publish step** — this repo ships no installable package artifact.
- New minimal `pyproject.toml` (doesn't exist today) containing only
  `[tool.semantic_release]` configuration — no `[project]` build metadata,
  since there is nothing to build/publish to a registry.
- Conventional-commit parsing (Angular preset): `feat:` → minor, `fix:` →
  patch, `BREAKING CHANGE:` footer → major. `docs:`/`chore:`/`ci:` do not
  trigger a release.
- **Version target:** `.claude-plugin/plugin.json`'s `"version"` field (today
  hardcoded `"0.1.0"`). `python-semantic-release` has no built-in JSON
  version-file writer, so use its `build_command` hook (the same mechanism
  `downbeat` used to resync `uv.lock`): a small script,
  `scripts/bump_plugin_version.py`, invoked by semantic-release with
  `NEW_VERSION` in the environment, that loads `.claude-plugin/plugin.json`,
  overwrites `version`, and rewrites the file preserving key order. The
  version-bump commit `git add`s this file alongside the auto-generated
  `CHANGELOG.md`.
- `CHANGELOG.md` (doesn't exist yet): auto-generated by semantic-release,
  Angular changelog template, committed as part of the version-bump commit.
- **Publish target:** GitHub Release only (`python-semantic-release publish`
  with `upload_to_vcs_release: true`, no PyPI/npm upload config at all). The
  release body is the changelog section for that version.

## 2. History anchoring

`python-semantic-release` parses the full commit history by default to infer
the current/next version. To keep the old non-conventional `[R2]`/`[R3]`
commits from being misclassified: before wiring semantic-release, tag the
current `main` HEAD (`e2c2920`) as `v0.1.0` manually
(`git tag v0.1.0 e2c2920 && git push origin v0.1.0`). Semantic-release then
only considers commits after this tag when computing the next version — the
first automated release will be `v0.1.1`/`v0.2.0`/etc. depending on what
`feat`/`fix` commits land after this point, cleanly ignoring history before
the anchor.

## 3. CI: `release.yml`

New file `.github/workflows/release.yml`:

- **Trigger:** `workflow_run` on `portability.yml` completing, filtered to
  `github.event.workflow_run.conclusion == 'success'` and
  `github.event.workflow_run.head_branch == 'main'` — identical trigger shape
  to `downbeat`'s `release.yml`.
- **Permissions:** `contents: write` (needed to push the tag + version-bump
  commit + create the Release).
- **Auth:** default `GITHUB_TOKEN` — **not** a PAT. Verified live
  (`gh api repos/FreddieMcHeart/claude-core/rulesets` /
  `.../branches/main/protection`, both `403 Upgrade to GitHub Pro or make
  this repository public to enable this feature`) that branch
  rulesets/protection are unavailable on a private free-tier repo — so the
  `github-actions[bot]`-vs-ruleset-bypass problem hit on `downbeat` cannot
  occur here today. Add an inline comment in the workflow flagging that this
  must be revisited (PAT via `RELEASE_TOKEN` secret) if/when the repo goes
  public and a ruleset with required status checks is added on `main`.
- **Steps:** checkout with `fetch-depth: 0` (semantic-release needs full tag
  history), install `python-semantic-release` (via `pip install
  python-semantic-release`, matching the repo's existing non-uv, plain-pip CI
  style from `portability.yml`), run `semantic-release version` then
  `semantic-release publish`.

## 4. README rewrite

Full `downbeat`-style structure:

1. **Title + badges:** CI status
   (`img.shields.io/github/actions/workflow/status/FreddieMcHeart/claude-core/portability.yml?branch=main`),
   latest release
   (`img.shields.io/github/v/release/FreddieMcHeart/claude-core`). No license
   badge yet — deferred with the LICENSE file itself to the "go public" task.
2. **Hook paragraph:** what claude-core is and why — portable,
   project-agnostic Claude Code operating manual, no Mondu-specific content,
   versioned via semantic-release.
3. **Real terminal demo:** an actual captured run of `bootstrap.sh` +
   `doctor.sh` on a real machine — recorded via VHS if available in this
   environment, otherwise a plain captured terminal transcript
   (`examples/bootstrap-and-doctor/demo.txt` or equivalent). Never fabricate
   sample output — decide the exact capture mechanism at implementation time
   based on what's actually installable in this environment.
4. **Install section:** existing `git clone` + `bootstrap.sh` snippet, plus a
   new subsection for the native plugin path (`claude plugin install
   ~/dev/claude-core/.claude-plugin`) including the `--migrate-to-plugin`
   flag already shipped in Phase 2.
5. **Release process section:** one short paragraph — semantic-release,
   conventional commits trigger releases, link to `CHANGELOG.md`.
6. **Contents section (expanded):** `models-router`, `delegation-discipline`,
   `claude-cost-audit` skills, plus the `claude-core-hooks` native plugin
   (cost-discipline hook, Phase-2 migration).

## Testing

- `scripts/bump_plugin_version.py` gets a unit test
  (`tests/test_bump_plugin_version.py`) verifying it rewrites only the
  `version` key and preserves the rest of `plugin.json` byte-for-byte
  otherwise.
- No automated test for `release.yml` itself (matches `downbeat`'s approach —
  first real run is the verification gate, per the playbook).

## Out of scope (explicitly deferred)

- Making the repository public.
- LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT.md, SECURITY.md.
- Name-squatting checks (no PyPI/npm package name at stake here).
- PAT/`RELEASE_TOKEN` wiring for the ruleset-bypass problem (not reproducible
  today on a private repo — revisit when going public).
